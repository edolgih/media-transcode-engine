# Архитектурные заметки

## 1. Назначение
Этот документ фиксирует архитектурные ограничения для Core и прикладного уровня, а также единую политику по осям расширения.
Цель — предсказуемое развитие: добавлять функционал без неконтролируемого роста классов и утечки доменной логики за границы Core.

## 2. Текущие границы системы
- Прикладной уровень отвечает за синтаксический парсинг, транспорт аргументов и UX процесса.
- Core отвечает за доменные значения по умолчанию, валидацию, нормализацию и принятие решений.
- Внешние обертки прикладного уровня (CLI/PowerShell) не должны дублировать бизнес-правила Core.

## 3. Модель контрактов
- Внешний вход может быть частичным/невалидным.
- Доменный контракт должен быть валиден в момент создания.
- Доменные контракты неизменяемы снаружи (нет публичного пути мутации инвариантов).
- Допустимые значения и значения по умолчанию должны быть централизованы (единый источник истины).

## 4. Оси расширения
Ось расширения — это стабильное направление функционального развития, которое может эволюционировать независимо.

Согласованные оси для проекта:
- Поведение профиля совместимости.
- Целевой видеокодек.
- Целевой контейнер/упаковка.
- Целевое разрешение и политика downscale.
- Режим обращения с исходником и политика именования.
- Политика rate control и качества.
- Политика таймстемпов и аудио.

## 5. Влияние на текущий дизайн
- Доменная валидация остается в Core на уровне создания доменных объектов.
- Валидация на прикладном уровне остается только синтаксической.
- Если согласована новая ось, предпочтителен additive-подход:
  - добавить/расширить поля доменного контракта;
  - вводить точечные policy-классы только там, где действительно есть независимость;
  - избегать монолитного "один engine на всё" с разрастающимся ветвлением.

## 6. Протокол изменений
Для любой новой фичи:
1. Явно указать, к какой оси она относится.
2. Сначала определить доменный контракт и значения по умолчанию.
3. Выбрать стратегию по классам (расширить существующий или добавить новый) по архитектурным принципам этого документа.
4. Добавить тесты на границе владельца поведения:
   - Core-тесты поведения для бизнес-логики.
   - Контрактные тесты прикладного уровня/wrapper для маршрутизации и UX.
5. Обновить docs/help после фиксации поведения.

## 7. Не-цели
- Нет дублирования доменных правил во wrapper-слоях.
- Нет новых абстракций "на всякий случай".

## 8. ООП-ориентиры для общего Engine
- Доменный слой выражает инварианты через валидные при создании неизменяемые объекты.
- Правила принятия решений по независимым осям (кодек, контейнер, compute, downscale и т.д.) выносятся в отдельные объекты поведения.
- При наличии устойчивой оси изменений предпочтителен полиморфизм (стратегии/политики), а не рост условных ветвлений в одном классе.
- Условная маршрутизация допустима в основном на границах системы (парсинг входа, wiring/composition root), но не как основной механизм доменной логики.
- Оркестратор должен координировать взаимодействие объектов, а не накапливать бизнес-правила.
- Расширение системы должно быть преимущественно additive: новые варианты поведения добавляются новыми объектами по согласованной оси, без переписывания существующих правил.
